# 阅游文档与编码守卫

## 编码要求

- 所有 Markdown 使用 UTF-8。
- 中文标题、表格、列表和代码块必须能在编辑器与 GitHub 正常显示。
- 发现乱码时先确认文件真实编码，不盲目整篇重写。
- 只修复与当前任务相关的文档，不制造大面积无关 diff。

## DevelopmentPlan 规则

- 文件名格式：`YYYYMMDD_中文任务概要.md`。
- 每天只维护一个今日任务汇总文件。
- 标题要随任务内容更新，避免文件名和正文不一致。
- 任务内容应包含目标、变更点、验证结果、后续风险。

## development_log 规则

- 按日期倒序或沿用现有格式。
- 每条日志写清类型：功能、修复、重构、测试、文档、维护。
- 记录核心文件和验证命令。
- 不写"优化若干问题"这类不可追溯描述。

## README 更新规则

仅当以下内容变化时更新：

- 启动、构建、测试命令。
- 技术栈、状态管理、依赖版本。
- 核心功能或用户可见行为。
- 架构约束、TTS 契约、发布流程。

如果只是内部实现微调，优先更新日志，不动 README。
