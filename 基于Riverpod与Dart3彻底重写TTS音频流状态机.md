# 🛠 架构重构任务：基于 Riverpod 与 Dart 3 彻底重写 TTS 音频流状态机

## 1. 任务背景

项目级 Riverpod 迁移已正式完成。当前目标是针对 `features/audio` 模块，利用 Riverpod 响应式特性与 Dart 3 模式匹配，重构高内聚、低副作用的 TTS 音频流状态机。同时，需完美兼容与自定义 Go 语言 TTS 流媒体服务器的 HTTP 交互。

## 2. 核心执行要求

### 2.1 密封状态定义 (Sealed State)

- 使用 Dart 3 的 `sealed class` 定义严格的音频状态树（如 `TtsAudioState`）。
- 状态节点必须穷尽音频流的生命周期，至少包含：
  - `Idle` (空闲/未初始化)
  - `Buffering` (缓冲中：必须包含当前已缓冲句数/进度状态)
  - `Playing` (播放中：必须包含当前正在播放的文本索引及音频流句柄)
  - `Paused` (已暂停)
  - `Error` (异常中断：必须包含错误类型与脱敏后的日志信息)

### 2.2 状态机接管 (Riverpod Notifier)

- 创建 `TtsAudioNotifier` (建议继承自 `AsyncNotifier` 或依据流特性选用)。
- **核心逻辑迁移**：将现有的“默认 6 句缓冲”机制移入 Notifier 内部进行状态管理。实现严密的队列监控：当缓冲队列出现不足时，自动触发向 Go TTS 服务器的预加载请求。
- **资源管理**：利用 Riverpod 的生命周期钩子（如 `ref.onDispose`）确保底层音频播放器句柄和网络 Stream 得到 100% 释放，杜绝内存泄漏。

### 2.3 UI 消费与防劣化隔离

- 在 UI 层（例如提词器或播放面板）消费状态时，**强制**使用 Dart 3 的 `switch` 表达式处理 `TtsAudioState`，利用编译期的穷尽性检查（Exhaustiveness）消除未捕获状态的隐患。
- **红线警告**：严禁在 `build` 方法中夹杂任何触发网络请求、启动播放或暂停等副作用（Side Effects）操作，严格遵循单向数据流。

### 2.4 Isolate 并发调度与 Impeller 性能防劣化

- **线程隔离**：从 Go 语言 TTS 服务端拉取到的音频二进制流（Chunk）解析、拼接以及缓冲队列的容量计算，**绝对禁止**占用主 Isolate。必须剥离出独立的 `Isolate` 或使用 `compute` 函数进行后台线程处理。
- **满帧保障**：确保主线程仅负责音频播放句柄的调度和微秒级的状态通知。状态机向 UI 抛出 `Buffering` 或 `Playing` 状态时，需进行合理的节流（Throttling）或防抖（Debouncing）处理，严禁高频 `notify` 导致 UI 层的 `RepaintBoundary` 被击穿，必须死守 60FPS 渲染底线。

### 2.5 进度同步与绝对数据隐私

- **纯本地持久化**：当 `TtsAudioState` 随着音频流不断向前推进时，状态机内部需联动更新当前书籍的“阅读进度”。此进度的存储**仅限本地设备沙盒**。
- **隐私红线拦截**：严禁在状态机或任何拦截器中，将用户的“当前听书进度”、“个性化语音设置”等数据打包发送或同步至任何云端服务器（包括但不限于现有的崩溃上报或外部统计接口）。Sentry 上报中必须通过黑名单过滤掉这些字段。
