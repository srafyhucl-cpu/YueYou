## 三、大厂规范对比

### 3.1 行业通用标准

| 规范维度 | 大厂推荐值 | 来源/依据 |
|----------|-----------|----------|
| 单文件行数上限 | **300-500 行** | Google Java Style Guide、Clean Code 原则 |
| 单类方法数上限 | **20 个以内** | SOLID 单一职责原则 |
| 单类行数上限 | **200-300 行** | Robert C. Martin《Clean Code》 |
| 单方法行数上限 | **20-30 行** | Clean Code 原则 |

> **注意**：Flutter/Dart 生态中，UI 组件因包含 build 方法、动画配置等声明式代码，天然会比纯逻辑类更大。Effective Dart 官方指南并未规定硬性行数限制，但强调"CONSIDER declaring multiple classes in the same library"——即允许同文件多类，但应保持职责内聚。

### 3.2 Flutter 特殊性说明

Flutter 中以下类型的文件行数偏大是**行业常见现象**，不一定需要拆分：
- **动画密集型组件**（如 `board_mascot.dart` 730 行，7 个 AnimationController）
- **复杂表单/设置页面**（如 `settings_screen.dart` 756 行，11 个私有子 Widget）
- **主页面/仪表盘**（如 `dashboard_screen.dart` 633 行，聚合多个子模块）

但以下类型的大文件通常需要重构：
- **服务类**（Service）超过 300 行 → 职责过多
- **状态管理类**（Provider/Notifier）超过 400 行 → 状态逻辑过于复杂
- **单文件多类且无明确关联** → 应拆分为独立文件

### 3.3 项目评估结论

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构分层 | ⭐⭐⭐⭐⭐ | Feature-Driven Clean Architecture，层级边界清晰 |
| 状态管理 | ⭐⭐⭐⭐ | Riverpod 使用规范，Provider 按功能模块组织 |
| 关注点分离 | ⭐⭐⭐⭐ | 大部分 UI 与业务逻辑分离良好，个别处有泄漏 |
| 文件大小控制 | ⭐⭐⭐ | 存在 1 个严重上帝类（1266 行），3 个偏大的 Provider |
| 测试覆盖 | ⭐⭐⭐⭐ | 总体 57.66%，核心模块覆盖较好，CI 门禁严格 |
| 依赖管理 | ⭐⭐⭐⭐⭐ | 依赖精简，无冗余 |
| 工程化 | ⭐⭐⭐⭐⭐ | CI/CD、AI 代码检查、11 个 Agent 技能定义、24 份开发计划 |

**总体评价**：项目整体架构质量**高于一般 Flutter 中小型项目**，大文件问题集中在少数几个文件，属于可控范围。

## 四、改进建议（按优先级排序）

### P0：`tts_engine_service.dart`（1266 行）—— 必须重构

**问题**：单文件 14 个类，核心类 7+ 职责，是全项目最严重的上帝类。

**拆分方案**：

| 新文件 | 内容 | 预估行数 |
|--------|------|----------|
| `audio/domain/tts_interfaces.dart` | 抽象接口：`TtsAudioPlayer`、`TtsWakeLock`、`TtsFallbackEngine`、`TtsHttpClient`、`HttpClientInterface` | ~80 行 |
| `audio/domain/tts_http_models.dart` | 数据模型：`TtsHttpResponse`、`_TtsHttpStatusException`、`TtsPlaybackState` 枚举 | ~60 行 |
| `audio/services/tts_http_client.dart` | HTTP 实现：`_RealHttpClient`、`_RealTtsHttpClient` | ~150 行 |
| `audio/services/tts_audio_adapters.dart` | 适配器：`_RealAudioPlayer`、`_RealWakeLock`、`_FlutterTtsFallbackEngine` | ~200 行 |
| `audio/services/tts_engine_service.dart` | 核心引擎（精简后） | ~400 行 |

### P1：`tts_audio_notifier.dart`（728 行）—— 建议重构

**问题**：单类 37+ 方法，双轨状态机 + 降级逻辑耦合。

**拆分方案**：
- 将降级逻辑（`_degradeToLocal`、`_pumpDegraded`）抽取为 `audio/services/tts_degradation_strategy.dart`
- 预估可减少 ~150 行

### P1：`reader_provider.dart`（638 行）—— 建议重构

**问题**：单类 34+ 方法，文本管理与 TTS 协调混合。

**拆分方案**：
- 将 `TtsSentenceSource` 实现抽取为 `reader/domain/sentence_source.dart`
- 将分章懒加载抽取为 `reader/services/chapter_loader.dart`
- 预估可减少 ~200 行

### P2：`settings_screen.dart`（756 行）—— 可选优化

**问题**：UI 子组件过多（11 个私有 Widget）。

**拆分方案**：
- 将 `_TtsTestButton` + `_TtsTestResultDialog`（~200 行）抽取为 `tts_test_widget.dart`
- 预估可减少 ~200 行

### P2：`square_board.dart`（783 行）—— 可选优化

**问题**：分享功能和游戏结束对话框混在棋盘组件中。

**拆分方案**：
- 将分享逻辑和游戏结束对话框抽取为独立组件
- 预估可减少 ~150 行

### P3：以下文件属于 Flutter 常见模式，**不建议优先重构**

| 文件 | 行数 | 原因 |
|------|------|------|
| `board_mascot.dart` | 730 | 动画密集型组件，7 个 AnimationController，拆分收益低 |
| `dashboard_screen.dart` | 633 | 主页面聚合层，属于正常模式 |
| `game_provider.dart` | 485 | 游戏状态机逻辑，行数尚可接受 |
| `storage_service.dart` | 401 | 统一存储服务，职责明确 |

## 五、额外发现的问题

### 5.1 StorageService 全局静态单例（中度风险）

- 使用全静态方法模式，在 8 个文件中被直接调用（共 110 处引用）
- UI 层（如 `library_screen.dart`）直接调用 `StorageService.getCurrentChapterIndex()`，跨越了架构层
- **建议**：逐步迁移为 Riverpod Provider 依赖注入

### 5.2 Riverpod Provider 类型不一致（低风险）

- 4 个 Provider 使用 `ChangeNotifierProvider`，1 个使用 `NotifierProvider`
- 项目已有 Riverpod 迁移计划，建议按计划推进

### 5.3 Linting 严格度可提升

- `strict-inference`、`strict-casts`、`strict-raw-types` 均设为 `false`
- 建议逐步开启以提升类型安全

## 六、给开发小白的总结

### 你的项目到底怎么样？

**好消息**：你的项目架构做得相当不错！
- ✅ 采用了大厂级别的 Feature-Driven Clean Architecture
- ✅ 有完善的 CI/CD 流水线和代码质量门禁
- ✅ 测试覆盖率在同类项目中属于中上水平
- ✅ 依赖管理精简，没有冗余

**需要改进的地方**：
- ⚠️ 有 1 个"超大文件"（`tts_engine_service.dart`，1266 行）需要拆分
- ⚠️ 有 2-3 个偏大的 Provider 文件建议优化
- ⚠️ StorageService 的全局静态模式建议改为依赖注入

### 真实大厂项目也是这样吗？

**大厂项目也会有几百行的文件**，尤其是 Flutter UI 组件。但大厂通常会有以下规范：
1. **单个类不超过 300 行**（你的 `tts_engine_service.dart` 超标 4 倍）
2. **单个文件只放一个公开类**（你的 `tts_engine_service.dart` 放了 14 个）
3. **Service 层文件不超过 300-400 行**（你的多个 Service 超标）
4. **Provider/Notifier 不超过 400 行**（你的有 3 个超标）

### 建议的行动顺序

1. **先别急着重构**——当前代码能跑、有测试，这是最重要的
2. **优先处理 P0**（`tts_engine_service.dart`），因为它是最大的技术债
3. **P1 按需处理**，在功能迭代时顺手拆分
4. **P2/P3 不用管**，属于 Flutter 正常现象
