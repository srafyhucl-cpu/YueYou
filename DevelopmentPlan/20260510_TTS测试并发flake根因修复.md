# 20260510 · TTS 测试并发 flake 根因修复

## 背景

`flutter test`（默认并行）下 `tts_audio_notifier_test.dart` 中的
`T-A pause → resume 必须重播暂停时的同一句` 等测试间歇性失败：

```text
Expected: <0>
  Actual: <1>
暂停 → onTtsItemFinished 绝不能被调用，否则 cursor 已跳
```

`flutter test --concurrency=1` 串行运行时 100% 通过，仅在并行 + CPU 竞争场景出现，
表明是典型的并发竞争 flake。

## 根因定位

`@/lib/features/audio/services/tts_engine_service.dart` 中：

1. `TtsEngineService` 构造函数把 `_initFuture = _initTtsHardware()` 设为
   **fire-and-forget**，未在调用 `downloadAudio` 前 await。
2. `_initTtsHardware` 内的 `_cleanupOrphanedTtsFiles` 会扫描 `getTemporaryDirectory()`
   并 **删除全部 `tts_*.mp3` 文件**（仅跳过 `_lastGeneratedAudioPath` 一项）。
3. 测试环境下 `getTemporaryDirectory()` 被 mock 为 `.`（项目根），多个并行测试
   isolate 共享该目录；每个 isolate 的 `_cleanupOrphanedTtsFiles` 会**误删
   其他 isolate 正在使用的下载产物**。
4. `_runPlayItem → playFile` 读到 `!await file.exists()` 路径，提前 `onComplete?.call()`，
   此时 `_isPausing == false`、`_pausedInterruptItemId == null`，pause 守卫无法生效，
   `_onPlaybackComplete` 直落 `onTtsItemFinished`，导致 `finishedCalls = 1`。

## 修复方案（lib 侧最小改动）

`@/lib/features/audio/services/tts_engine_service.dart`

1. **`downloadAudio` 顶部 `await _initFuture`**：单 isolate 场景下，确保
   `_cleanupOrphanedTtsFiles` 完成后再写入下载产物，杜绝同 isolate 内的
   清理 vs 写入并发。
2. **`_cleanupOrphanedTtsFiles` 跳过 60s 内修改过的文件**：跨 isolate / 跨进程
   场景下，活跃下载窗口内的文件不会被误删；只清理上一次 App 运行残留的
   真正"孤儿"文件。`stat` 失败时保守跳过。

两处合计 +19 行，零运行时副作用：

- 生产环境：init 通常在 play 前已完成，`await _initFuture` 立即返回；
  60s 窗口对真正孤儿（来自上次 App 运行）零误伤。
- 测试环境：彻底消除并发 flake，无需修改任何测试用例。

## 验证

- `flutter analyze`：零错误零警告。
- `flutter test`（默认并行）连跑 8 次：**8/8 全部通过**（664 +4 skipped）。
  之前同样 8 次约 50% flake 率（T-A pause 多次失败）。
