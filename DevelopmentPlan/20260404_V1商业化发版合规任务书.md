# 阅游 (YueYou) V1.0 商业化发版冲刺指南 (Production Release Sprint)

**项目背景**：经过前几轮的深度重构，「阅游」APP 的核心架构（Isolate 解析、双轨 TTS 引擎、赛博视觉系统）已极具稳定性。当前阶段目标是从“内测 Demo”正式蜕变为**“可上架各大应用市场（App Store/安卓商店）的 V1.0 商业化产品”**。
**当前痛点**：远端 TTS 服务器（47.94.102.250）正在联调中，目前应用存在单点故障风险（断网或远端服务异常时听书功能瘫痪），且缺乏上架必备的隐私合规机制。

**执行要求**：请作为大厂资深架构师，严格按照以下 P0/P1 优先级依次执行任务。每次只执行一个 Task，完成后向我汇报，等待我确认后再进行下一个。涉及 UI 时必须使用 CyberDimensions、CyberColors 等现有 Token。

---

## Task 1: 网络安全策略与 TTS 离线引擎降级兜底 (P0 - 核心保卫战)

**目标**：解决新服务器 HTTP 明文拦截问题，并引入系统自带的离线 TTS 作为终极兜底，确保“听书体验永远不断”。

1. **解除 HTTP 明文封杀**：
   - 修改 `android/app/src/main/AndroidManifest.xml`，在 `<application>` 标签中添加 `android:usesCleartextTraffic="true"`。
   - 修改 `ios/Runner/Info.plist`，添加 `NSAppTransportSecurity` 字典，配置 `NSAllowsArbitraryLoads` 为 `<true/>`。
2. **引入原生离线 TTS**：
   - 在 `pubspec.yaml` 中引入官方插件 `flutter_tts`。
3. **重构 TtsEngineService 降级逻辑**：
   - 在 `lib/features/audio/services/tts_engine_service.dart` 中，初始化 `FlutterTts` 实例。
   - 修改 `_downloadTtsAudio` 或 `_startPlayLoop` 逻辑：当远端 HTTP 请求发生超时（TimeoutException）、500 错误或解码失败时，**自动触发降级机制（Fallback）**，调用 `flutter_tts` 直接朗读当前句子的纯文本。
   - 触发降级时，通过 UI 抛出一个赛博风格的 Toast 提示：“云端神经网断开，已自动切换至本地仿生发声模块”。

## Task 2: 隐私合规与权限前置拦截 (P0 - 上架强制要求)

**目标**：满足国内/国际应用商店对隐私合规的严苛要求。

1. **赛博朋克风《隐私协议》弹窗**：
   - 创建 `lib/features/settings/presentation/widgets/privacy_agreement_modal.dart`。
   - 复用 `showCyberModal` 和 `CyberDimensions` 构建弹窗，包含“同意”与“拒绝并退出”按钮。
   - 逻辑：在 `main.dart` 或 `DashboardScreen` 初始化时，检查 `SharedPreferences` 中的 `hasAgreedPrivacy` 标志。若未同意，强制拦截在主界面之上，且不允许任何底层文件读取或网络请求发生。
2. **敏感权限前置说明（FilePicker）**：
   - 在 `lib/features/library/services/file_import_service.dart` 或其调用方（如 `CyberImportButton`），在真正调用系统文件选择器前，弹出一个轻量级的说明框：“阅游需要读取您的本地存储以解析数据芯片 (TXT文件)，是否授权？”。

## Task 3: 大文件内存硬阻断 (P0 - 极端设备保护)

**目标**：虽然采用了流式读取，但 V1.0 暂未实现数据库分页，需从入口处彻底阻断超大文件撑爆内存。

1. **文件大小硬性校验**：
   - 在 `lib/features/library/services/file_import_service.dart` 中，用户选中文件后，立即检查 `File.lengthSync()`。
   - 如果文件大小超过 **15MB**（可配置为常量），直接返回一个特定的错误对象或抛出异常。
   - UI 层捕获后，提示：“V1.0 神经接驳器带宽有限，暂不支持超过 15MB 的超大型数据芯片，请分割后导入”。

## Task 4: 基础崩溃监控与热更新锚点 (P1 - 生产可观测性)

**目标**：为后续的快速迭代与数据驱动打好基建（暂不强依赖第三方 SDK，先做好代码占位）。

1. **全局错误捕获锚点 (Global Error Handler)**：
   - 在 `main.dart` 中，使用 `FlutterError.onError` 和 `PlatformDispatcher.instance.onError` 捕获全局未处理的红屏 and 异步崩溃。
   - 创建 `lib/core/utils/cyber_logger.dart`，将捕获到的堆栈信息连同设备上下文进行格式化输出（暂为 `debugPrint`，并在注释中标注：TODO: V1.1 接入 Sentry/Crashlytics）。
2. **版本更新检测桩 (Update Checker Stub)**：
   - 在 `DashboardScreen` 的 `initState` 中，增加一个异步方法 `_checkAppUpdates()`。
   - 暂时将其实现为一个空的 `Future`，但在注释中写明后续在此处调用远端 JSON 比对版本号，并预留调用 `showCyberConfirmDialog` 提示用户更新的逻辑位。
