## 十、思想纠偏复盘（PR-A~E 完成后追加）

### 偏差识别

PR-A~E 在执行过程中，**虽然产出大部分有真实开发规范价值**，但表述与动机
存在明显的「为合规而合规」偏差：

- 反复以「跌出警戒线 / AI 门禁 warning 消失 / ≤ N 行」作为 PR 成功标志
- review 模板把「行数变化」列在首位，「职责边界变化」未被强制论证
- PR-D 抽出 `audioStateToEngineState` / `snapshotOf` 等 inline 完全合理的小函数，
  主要动机是「再砍几行让 notifier 跌出警戒线 700」，而非「这两个函数独立后更
  利于复用 / 测试」
- PR-D 给 `TtsFallbackController` 设计了 13 个 callback 接口，是「为追求零反向
  引用」的过度严格——传 notifier 进去做 weak ref / typed facade 更简洁
- 工作流命名 `large-file-refactor-review` 和 skill 命名 `yueyou-file-size-guard`
  把「大文件 / file-size」当治理对象，而真正治理对象是「职责膨胀 / 上帝类反模式」

### 产物诚实评估（12 个抽出物）

**✅ 真规范驱动（10 个）**：

- `tts_engine_interfaces` / `tts_network_interfaces` —— 依赖反转基础
- `tts_http_models` —— 数据模型归 domain
- `tts_audio_adapters` —— 适配器层独立可换可测
- `tts_http_client` —— IO 实现独立
- `tts_audio_janitor` —— 无状态副作用函数
- `tts_diagnostics_service` —— 连接诊断独立可 mock
- `tts_audio_downloader` —— HTTP 流水线 + 本地降级流水线独立
- `tts_paused_interrupt_guard` —— 语义完整的有状态单元
- `reader_sentence_cursor` —— 核心算法独立可大量 edge case 喂测试

**⚠️ 价值有但代价偏高（1 个）**：

- `tts_fallback_controller` —— 13 callback 是「为追求零反向引用」的过度严格副产物

**⚠️ 边缘 / 凑行数（1 个）**：

- `tts_audio_state_helpers` —— `copyStateWithRate` 有价值；`audioStateToEngineState` 与 `snapshotOf` 两个 inline 在用方旁也合理

**结论**：12 个产物中 10 个真有规范价值（83%），2 个边缘 / 过度工程。整体方向正确，但部分细节动机偏了。

### 决策：不回滚

回滚成本（破坏现有依赖关系、测试 import 调整、commit 历史污染）> 边缘价值的
负面影响。但**记录在案**，明确：

- `tts_fallback_controller` 的 13 callback **不应作为后续抽离的样板**；未来如果
  需要做单元测试，可重构为 typed facade 注入。
- `audioStateToEngineState` / `snapshotOf` 在 helpers 文件的归类合理，但**类似
  动机不应再驱动新的抽离**。

### 入口纠偏（已落地）

为防止类似偏差复发，已修订 3 个 active 入口：

- **`.windsurf/rules/AGENT.md` 红线第 8 条**：从「单文件体量门禁」改为

  自检」、「反模式警告」（驳回为压行数而拆的动机）。

  - 顶部新增「核心原则：规范优先于合规」段落，明确「保留 warning 不动」也是

  - 1.2 节细化：黄灯 = 必须先评估、不是闭眼拆

### 后续行为准则

- 触发警戒线时，**先评估职责**，再决定动手——不动也是合法结论
