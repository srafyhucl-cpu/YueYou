## 二、TTS 引擎异常恢复机制修复

### 问题分析

`TtsEngineService._startPlayLoop` 中，`audioPlayer.setSource` 抛出异常时：

- 捕获了异常，但恢复流程未检查引擎生命周期状态（`_disposed`、`isEnabled`、`_loopSession`）。
- 引擎被禁用/销毁后，异常恢复仍会触发降级逻辑，引发测试时序竞争。

### 解决方案

在 `TimeoutException` 和 `catch(e)` 异常恢复分支中，调用 `_audioPlayer.stop()` 后立即加入生命周期守卫：

```dart
if (_disposed || !isEnabled || _loopSession != sessionAtStep) return;
```

确保异常时正确中断流程，不执行多余熔断与降级通知。

### 涉及文件

- `lib/features/audio/services/tts_engine_service.dart`

## 四、CI 覆盖率门控加固（task.md §2.2）

### 修改文件

`.github/workflows/flutter-ci.yml`

### 修改内容

| 步骤 | 修改前 | 修改后 |
| --- | --- | --- |
| `flutter analyze` 失败 | 直接 exit，无摘要 | 写入 `GITHUB_STEP_SUMMARY` 中文错误说明后 exit 1 |
| 业务核心覆盖率 < 90% | ⚠️ 仅警告，**不阻断 CI** | ❌ **exit 1 强制失败，禁止合并** |
| 覆盖率达标 | 简单打印 | 输出 `✅ 覆盖率达标` 摘要 |

### 强制阻断逻辑

```yaml
# 覆盖率未达标 → 强制 exit 1
if [ "$PASS" != "yes" ]; then
  echo "## ❌ 覆盖率未达标 — 本次 CI 失败，禁止合并" >> $GITHUB_STEP_SUMMARY
  exit 1
fi
```

> **注意**：GitHub Branch Protection Rules 需在仓库设置中手动开启，
> 将 `Flutter 持续集成 / 分析与测试` 添加为 Required Status Check，
> 才能在 GitHub 层面强制阻止未通过 CI 的 PR 合并。
