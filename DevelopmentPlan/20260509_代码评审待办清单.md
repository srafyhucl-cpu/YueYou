## ✅ P1 · `SettingsProvider` listener prev==next dead code（已修复 commit pending）

- **位置**：`@/lib/features/audio/providers/tts_audio_notifier.dart:95-99`
- **现象**：

  ```dart
  ref.listen(settingsProvider, (prev, next) {
    if (prev?.idleTimeout != next.idleTimeout) {
      _resetIdleTimer();
    }
  });
  ```

  `settingsProvider` 是 `ChangeNotifierProvider<SettingsProvider>`，Riverpod
  在 ChangeNotifier 通知时把同一个 `SettingsProvider` 实例既作为 `prev` 也作为
  `next` 传入。`prev?.idleTimeout != next.idleTimeout` 永远是 `false != false`
  即永远 `false`，分支**永远不会触发**。
- **影响**：用户在设置页修改静默暂停时长后，TTS Notifier 的 idleTimer **不会
  自动重新计算**。当前能工作只是因为 `_playNext` / `play` / 引擎心跳路径会
  顺带触发 `_resetIdleTimer`，掩盖了缺陷。
- **建议**：要么换为快照对比（保存上一次 `idleTimeout` 数值字段），要么改用
  `settingsProvider.select((s) => s.idleTimeout)` 让 Riverpod 帮忙拆分依赖。
- **回归覆盖**：修复后必须新增 fakeAsync 用例验证「`settings.setIdleTimeout(N)`
  必须触发 `_resetIdleTimer`」。

## ✅ P2 · `_refillBuffer` 内 `_degradeToLocal` fire-and-forget 易引发竞态（已修复）

- **位置**：`@/lib/features/audio/providers/tts_audio_notifier.dart:393-404`
- **现象**：

  ```dart
  if (_consecutiveFailures >= threshold) {
    CyberLogger.captureWarning(...);
    _degradeToLocal(request);  // 未 await
  }
  return;
  ```

  `_degradeToLocal` 是 `Future<void>`，调用方未 `await`。该函数内部多个
  `await _engine.stopAudio()` / `pauseAudio()` 在 microtask 完成前
  `_isDegradedToLocal=true` 还没同步生效。期间 `_prefetchRunner` 下一轮 loop
  仍可能调用 `_refillBuffer`，造成 `_consecutiveFailures` 继续累加，多次
  `_degradeToLocal` 重复触发。
- **影响**：测试场景下能观察到 `nextCalls` 不可精确预测；生产环境下可能产生
  重复降级日志、重复 `speakWithLocalTts` 调用，浪费 CPU。
- **建议**：把 `_isDegradedToLocal=true` 提升为 `_degradeToLocal` 的**同步**第
  一步（在第一个 `await` 之前），或在 `_refillBuffer` 触发降级后立即同步设置
  标志位。

## ✅ P3 · `TtsEngineService.cycleSpeed` 直接调 `setPlaybackRate` 不走安全包装（已修复）

- **位置**：`@/lib/features/audio/services/tts_engine_service.dart:763-770`
- **现象**：

  ```dart
  void cycleSpeed() {
    ...
    _audioPlayer.setPlaybackRate(_playbackRate);  // 不走 _safeSetPlaybackRate
    _settings.setTtsRate(_playbackRate);
    notifyListeners();
  }
  ```

  其他路径（如 `_syncSettingsInternal`）通过 `_safeSetPlaybackRate` 包装，失败
  会走 `_setLastError + captureWarning`。`cycleSpeed` 直接调，失败会向上抛。
- **影响**：用户切倍速时若播放器底层故障，错误不会被脱敏入库，反而向上层
  StatefulWidget 抛出导致 setState 红屏。
- **建议**：统一走 `_safeSetPlaybackRate(_playbackRate)`，与 `_syncSettingsInternal`
  保持容错一致。

## ✅ P3 · `DefaultBookService.getCatalog` 网络成功但无 chapters 字段未校验（已修复）

- **位置**：`@/lib/features/library/services/default_book_service.dart:51-74`
- **现象**：响应 `body['status'] == 'success'` 后直接 `body['chapters'] as List`。
  若服务端返回 success 但 `chapters` 字段缺失或 null，会抛 `_TypeError`，被外层
  catch 吞下走降级路径。
- **影响**：错误日志含义模糊（看起来像网络异常，实际是协议字段错）。
- **修复**：抽出 `_fallbackBuiltinCatalog()` 辅助方法；显式 `if (rawChapters is! List)`
  校验失败 → `captureWarning('协议字段缺失 chapters')` + `_fallbackBuiltinCatalog()`。
- **回归**：新增 2 条 default_book_service_test.dart 用例：chapters 字段缺失 / 为
  null 时必须降级返回内置 100 章。

## ✅ P3 · `_AlwaysFail500HttpClient` 测试中 download 路径未实现（已修复）

- **位置**：`@/test/features/audio/tts_audio_notifier_test.dart:92-113`
- **现象**：测试用 fake httpClient 仅实现 `post` 返回 500，`download` 方法是
  空实现（`async {}`）。生产 lib 中如果有路径只调 `download`，不会被这个
  fake 模拟到，测试盲区。
- **修复**：`download` 同步抛 `HttpException('simulated outage on download')`
  并增加 `downloadCalls` 计数，与 post 失败语义一致。
- **影响**：覆盖率工具可正确探测 download 失败路径。

## ✅ P3 · 测试套件中多处重复手工初始化 path_provider mock（已修复）

- **位置**：`@/test/utils/test_utils.dart:140-174`
- **现象**：`initializeTestEnvironment()` 已经 mock 了 path_provider 返回 `'.'`，
  但要让每个用例独立 temp dir 又得手动重写 channel。这套样板出现在多个测试
  文件里。
- **修复**：在 `test/utils/test_utils.dart` 新增工具方法
  `initializeTestEnvironmentWithIsolatedTempDir(prefix)`，内部统一执行：
  1. `initializeTestEnvironment()` 基础初始化
  2. `StorageService.resetForTesting() + init()` 隔离持久层
  3. `Directory.systemTemp.createTemp(prefix)` 创建独立目录
  4. 重写 path_provider channel 指向该目录
  5. 返回 tempDir 供 tearDown 清理使用。

## ✅ P3 · `DefaultBookService` 异步缓存写入失败仅记 warning，无重试（已修复）

- **位置**：`@/lib/features/library/services/default_book_service.dart:32-54`
- **现象**：`StorageService.saveBookCatalog().catchError(...)` 只记录 warning，
  不重试。下次 App 启动时仍然没缓存，又会发起网络拉取。
- **修复**：新增进程内内存缓存 `_catalogMemCache` + `_chapterMemCache`，
  优先级：内存 → 磁盘 → 网络。磁盘命中后 seed 内存；网络成功后先写内存再
  fire-and-forget 写盘（即使写盘失败本会话仍可复用）；**降级到内置常量时
  不写内存缓存**，保留下次重试网络的机会。同时提供
  `@visibleForTesting void clearMemoryCacheForTesting()`。
- **回归**：新增 6 条 default_book_service_test.dart 用例（getCatalog 内存
  命中 / 内存优先磁盘 / 降级不污染 / fetchChapter 内存命中 / isChapterCached
  内存命中 / clearMemoryCacheForTesting 双层清空）。
