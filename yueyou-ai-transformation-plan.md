# 阅游 AI 化改造方案：从小说阅读器到 AI 原生应用

将阅游从纯 TTS 小说阅读器改造为集成 LLM 对话、RAG 知识库、智能内容生成的 AI 产品，支撑"转大模型开发"简历。

---

## 一、架构变更

```
当前:  Flutter ←→ Go (Gin) ←→ edge-tts CLI ←→ OSS/CDN
目标:  Flutter ←→ Python (FastAPI) ←→ LLM API (DeepSeek/通义千问)
                         ├── ChromaDB (向量库)
                         ├── edge-tts CLI (TTS，保留)
                         └── OSS/CDN (存储，保留)
```

**为什么换 Python**：LLM 生态全部是 Python——LangChain、LlamaIndex、ChromaDB、OpenAI SDK 等。Go 在这个领域是二等公民，简历上写"Go 调 LLM"不如"Python 全栈 AI"有说服力。

---

## 二、四大 AI 功能模块

### 1. AI 角色扮演对话（核心亮点）
- 用户选择西游记角色（孙悟空、唐僧、猪八戒等），进入沉浸式对话
- 技术栈：FastAPI + DeepSeek/通义千问 Chat API + 对话记忆（Redis 或内存 LRU）
- 展示能力：Prompt Engineering、System Prompt 角色设定、多轮对话管理、Token 预算控制
- 亮点：角色人设一致性、引用原著情节、赛博朋克 UI 对话界面

### 2. RAG 智能问答（技术深度）
- "孙悟空大闹天宫在哪一章？""唐僧收了几个徒弟？"
- 技术栈：ChromaDB + 通义千问 Embedding + 章节分块 + 语义检索 + LLM 生成答案
- 展示能力：文档分块策略、向量检索、RAG Pipeline、引用溯源
- 数据源：已有的 100 章西游记章节文本（OSS 上）

### 3. 智能内容生成（广度展示）
- 章节 AI 摘要：读完一章自动生成摘要
- AI 续写：给定开头，生成符合西游记风格的续写
- 2048 策略提示：AI 分析当前棋盘给出下一步建议
- 展示能力：Few-shot Prompting、结构化输出、Function Calling

### 4. 原有功能保留
- TTS 朗读（edge-tts + OSS，不变）
- 2048 游戏（不变）
- 阅读进度本地存储（不变）

### 服务器费用

| 资源 | 当前 | 改造后 | 增量费用 |
|------|------|--------|---------|
| 云服务器 | 已有（部署 Go） | 同一台部署 Python | ¥0 |

## 六、风险与注意事项

- **edge-tts 依赖**：服务器上需保留 `edge-tts` Python 包，FastAPI 通过 `subprocess` 调用，与原 Go 方式一致
- **ChromaDB 持久化**：向量数据文件需加入备份策略
- **API Key 管理**：多个 LLM 的 Key 通过 .env 注入，严禁提交版本库
- **对话记忆上限**：需设置 Token 预算，超出时自动摘要压缩历史
