阅游性能架构改进方案 YUEYOU PERFORMANCE BLUEPRINT / 2026-07-13

管理摘要 / Executive Brief

源码审计显示，首要风险是主线程负担过重。

当前风险集中在启动前工作过多、动画反复布局、小变化触发整页刷新、整本书多份常驻内存和旧异步任务无法及时停止五条链路。建议拆为六个可独立验收、可单独回滚的性能改进包，先建立真机基线，再决定后续投入。

报告边界 本页数值均为建议门槛或待验证目标，不是当前实测收益。实际影响与最终结论必须由真机性能模式（Profile）数据确认。

目标门槛

把性能变成可验收的产品能力

先定义统一口径，再决定优化是否有效。高刷目标和基础目标必须分开，不再用单次 CPU 跑分代表整机体验。

基础流畅度 < 16.7ms 60Hz 设备：95% 的界面构建与图形绘制帧各自达标
高刷体验 < 8.3ms 120Hz 设备：95% 的界面构建与图形绘制帧各自达标
TTS 会话 1 + 1 最多 1 条预取轨 + 1 条播放轨
大书内存模型 有界窗口 只保留当前与相邻物理分片，不随整书线性常驻

为什么现在改

四类风险直接影响核心产品体验

阅游的主线是“听读 + 陪伴式 2048”。性能问题会同时放大首用流失、播放中断、低端机兼容和维护成本。

01

冷启动可用性

音频初始化、CPU 微基准和环境音生成靠近首帧执行。弱设备上可能表现为启动迟缓、首屏停顿或隐私流程不稳定。

02

听读与游戏并行

棋子位移动画进入布局阶段，提词器进度同时触发滚动和文本重建。两条热路径叠加时，更容易出现掉帧和功耗上升。

03

大书稳定性

15MB TXT 虽在 Isolate 解析，完整行列表仍回传主 Isolate，并整体 JSON 编解码。峰值内存与开书耗时随整书增长。

04

长会话可靠性

TTS runner 与切章请求缺少完整代际约束。快速暂停恢复或连续切章时，旧任务可能继续消耗网络、CPU 和磁盘。

改进范围

不做大爆炸重写，只替换五条关键链路

每项变更都有独立边界、自动化证据和回滚条件。现有视觉、TTS 两步下载契约和本地隐私原则保持不变。

管理决策

需要确认的不是技术细节，而是执行顺序

性能治理不能绕过现有 M0 安全与隐私、M1 交付可信度门禁。基线采集可以并行，业务架构改造按依赖映射进入 M2。

  • 首批投入先批准 PERF-0 基线采集 1-2 人日；基线评审后再按需批准剩余 16-22 人日，已达标链路直接取消。
  • 目标设备研发负责人指定至少一台中低端 Android 与一台 120Hz 设备，测试负责人维护场景与口径。
  • 范围纪律同意 2048 只调整渲染管线，不把 4 x 4 算法搬进 Isolate，也不重做视觉。
  • 迁移策略同意大书仓储由 feature flag 控制，首个稳定版本双读并 legacy shadow write；迁移包不删除旧数据。
粗粒度实施量级 17-24 人日

首批只批准 PERF-0 的 1-2 人日；其余 16-22 人日在基线评审后分段批准。总量不含 M0 安全修复，也不构成排期承诺。

工程详设 / Technical Design

按职责拆，不按行数拆；按证据改，不按感觉改。

以下模块是目标边界，不要求一次性全部创建。只有当新单元可独立测试、依赖方向更清晰且不会引入生命周期断链时，才进入拆分 PR。

强约束 TTS 继续遵守 POST 获取 JSON URL、GET 下载音频的两步契约；阅读正文与进度继续纯本地存储。

模块设计

九个清晰边界，覆盖启动到持久化

展开条目可查看职责、输入输出、依赖、失败处理与验收方式。默认先实现替换成本最低、证据最确定的边界。

BOOT-01 BootstrapCoordinator把首帧必需工作与非关键服务初始化彻底分开

职责与接口

读取本地隐私状态，选择 ConsentApp 或完整应用；完整应用首帧完成后调度音频、更新检查与可观测性服务。输出只包含启动阶段和 readiness，不承载业务状态。

依赖方向

位于应用 composition/bootstrap 边界，可依赖 core 基础设施和各 feature 的 provider factory；任何 feature 不得反向引用它。

失败与回滚

非关键服务初始化失败只记录脱敏错误并进入 degraded ready；隐私状态读取失败默认进入未同意路径。每项服务保持独立初始化和幂等释放。

测试与验收

  • 首次启动同意前网络请求严格为 0。
  • 已同意启动只创建一套播放器与 ProviderScope。
  • 首帧期间不执行 PCM 生成或 CPU 微基准。
FRAME-01 PerformanceGate + VisualEffectsPolicy用真实帧数据控制 blur、粒子、阴影和常驻动画

职责与接口

采集非内容型 FrameTiming，在稳定窗口计算 build/raster P95、P99 与掉帧率；输出不可变视觉策略，而不是把设备贴上永久的高、中、低标签。

控制范围

策略字段分别控制 backdrop blur、复杂阴影、粒子数量、雨幕、声纹刷新率和吉祥物闲置动画。消费者只 select 自己需要的字段。

迟滞策略

连续多个窗口超预算才降档，连续更长时间稳定才升档；切后台停止采样，低电量或系统降帧时不误判为永久低性能。

隐私与验收

只记录耗时、帧数、设备刷新率和场景编号，不记录正文、书名、章节或本地路径。60Hz 与 120Hz 使用不同门槛。

GAME-01 GameEngine + TileTransformLayer4 x 4 逻辑保持同步纯函数，移动动画只进入绘制阶段

领域结果

move(board, direction, random) 返回新棋盘、分数、合并事件和每个 tile 的 from/to 坐标。算法仅使用 Dart 核心库，不引入 Flutter，也不放进 Isolate。

渲染结构

静态网格、动态棋子、粒子效果和 Game Over 覆盖层分离。棋子拥有稳定尺寸与 key，通过 Transform.translate 消费坐标差。

状态投影

Dashboard 只订阅 score/combo 投影，棋盘只订阅 board revision，音效开关不进入棋盘渲染状态。16 格不可变复制成本可控。

测试与验收

  • 领域性质测试覆盖守恒、单次合并和顺序。
  • 连续 30 次滑动无持续 layout 抖动。
  • 是否给单 tile 增加 RepaintBoundary 由 Profile 决定。
READ-01 TeleprompterPainter句子只布局一次，音频进度只更新位移和裁剪

输入输出

输入为句子快照、播放状态和归一化进度；输出为已读/未读两层文本、中心线和横向偏移。句子变化才重新建立 TextPainter。

进度策略

音频 position 事件只更新目标 progress，由单一 vsync 平滑插值。字符高亮索引执行 distinct，避免相同字符区间重复绘制。

重绘边界

毛玻璃背景保持静态，文字画布和中心线作为独立 RepaintBoundary。错误浮层不进入进度 ticker。

测试与验收

同一句播放期间不触发全文 widget rebuild；进度 0、0.5、1.0 的像素与语义测试稳定；暂停时 ticker 归零。

STATE-01 StateProjection + AppComposition局部字段变化只唤醒真正依赖它的消费者

投影规则

为 Game、Reader、Settings、Bookshelf 定义小粒度不可变 view state 或 record selector。禁止 presentation watch 可变服务整对象后再挑字段。

跨模块联动

Settings 到 Ambient、Game 到 TTS 用户活动等联动由 composition 层 ref.listen 承担；Widget 只转发用户命令。

滑杆策略

拖动阶段只更新页面局部预览和节流后的播放器音量；onChangeEnd 提交持久化，失败时回退到最近已保存值并提示。

验收

游戏移动不重建 Dashboard 根；环境音音量变化不重建提词器；导入按钮只响应 isParsing 投影。

BOOK-01 BookContentRepository整书 JSON 改为 manifest、有界物理分片与按需窗口

存储格式

manifest.json 分别记录语义章节与物理 chunk 映射。物理 chunk 建议以 256KB 或 1000 句为上限，任一先到即切分；超长段落继续按句切分，避免“无章节书”退化为一个 15MB 文件。

导入流程

超过 100KB 的解析由 ImportJob 在 Isolate 内完成编码识别、索引和同目录 staging 原子发布，只返回小型 metadata。result/error/onExit 三个端口竞争完成同一 Completer；取消必须完成 Future、关闭全部端口并清理已知 .part。

读取与回滚

Reader 只持当前及相邻物理 chunk。首个稳定版本在 feature flag 下双读并 legacy shadow write，保证旧 APK 可读新导入书；迁移包不删除旧数据，至少稳定一个版本后另立清理 PR。

测试与验收

  • UTF-8、GBK、BOM、无章节 15MB 大书均形成有界 chunk。
  • 导入取消后 Future、Isolate、端口与 staging 文件全部收口。
  • 恢复内存随固定 chunk 窗口变化，不随整书线性增长。
TTS-01 TtsSessionController + BufferPolicy会话编排归 Provider，策略归 domain，I/O 归 service

会话模型

Provider/Notifier 独立拥有 semantic session、runnerEpoch、队列与播放状态；domain 只保存 BufferPolicy。暂停/恢复仅递增 runnerEpoch 并保留当前句与缓冲，换书、切章、换音色或 stop 才创建新 semantic session。

队列模型

播放取走一项时发送补充信号，预取完成时唤醒播放，不再用 300ms 至 2s 轮询。默认下载并发为 1，只有基线证明必要时才评估有界并发 2。

网络与文件

Service 只负责复用 HttpClient、两步下载、播放器与文件 I/O。每个下载输入独立 operationId 与 CancellationToken；ref.onDispose 先取消活动请求，再由资源 owner 执行 close(force: true)。

契约与验收

始终先 POST 解析 JSON URL，再 GET 音频并以类型化结果返回。快速 pause/play 20 次后 runner 上限不变、buffer 不超上限；切章后旧 semantic session 与旧 operation 均不得回写。

READ-02 ReaderLoadGeneration连续切章、恢复和自动推进统一采用 latest-wins

提交规则

每个加载意图递增 generation。fetch、parse、持久化前后均校验 generation 与 disposed；只有最新意图可以提交章节、正文和进度。

状态模型

加载状态同时保存 requestedChapter、committedChapter、generation 和错误类型，避免“UI 显示 B 章但正文来自 A 章”的半提交状态。

错误处理

过期请求静默丢弃结果并释放资源，不显示错误；最新请求失败才进入可重试状态。日志不包含正文。

测试

覆盖 A 慢 B 快、A 快 B 慢、加载中 dispose、恢复与手动切章竞争、章末自动推进与点击竞争。

DATA-01 SnapshotWriter游戏与阅读进度采用单写者、合并版本和生命周期 flush

写入模型

状态变更提交带 revision 的不可变快照，writer 串行写入并合并中间版本，只保证最新 revision 最终持久化。

触发策略

游戏保留 1 秒 debounce；阅读进度采用 debounce + maxWait 周期检查点。暂停、切章、退出前执行 best-effort flush，不把生命周期回调当成强持久化保证。

失败处理

写失败保留最新待写快照并记录一次脱敏 warning；下一次触发继续尝试，避免多个 fire-and-forget 写互相覆盖。磁盘持续失败时不承诺数据丢失上限。

验收

并发提交只产生串行磁盘写；旧 revision 不得覆盖新 revision；强杀与写失败通过故障注入记录实际损失窗口，不用 flush 推导绝对保证。

关键时序

TTS 切章后的安全提交链

Reader generation、TTS semantic session、runnerEpoch 与 HTTP operationId 必须由各自 owner 独立递增。模块之间只传不可变的 contentRevision，不共享计数器，也不承诺跨 Provider 原子事务。

  1. 用户选择 B 章Reader generation + 1，仅使旧内容加载失效
  2. 停止旧音频任务TTS 独立更新 semantic session 与 runnerEpoch，并取消 HTTP operation
  3. 读取与解析Repository 获取 B 章；超过 100KB 的解析进入 Isolate
  4. Reader 单次提交generation 与 disposed 校验后提交不可变 ChapterSnapshot
  5. Composition 启动 TTS观察 contentRevision，创建绑定该版本的新 TTS 会话

跨层只传小对象、不可变快照和明确结果

下面是建议的逻辑接口，不要求名称逐字一致。实现时优先复用现有 domain models，避免为了架构感重复建模。

实施路线 / Delivery Plan

先量化，再止损，再重构；任何阶段都能停。

推荐拆为六个性能改进包并分别形成独立 PR。PERF-0 只建立基线；PERF-1 至 PERF-3 优先处理高确定性问题；PERF-4 的 TTS 生命周期与 PERF-5 的数据迁移只有通过专项门禁后才实施。

进入条件 业务改造不得绕过现有 M0 安全与隐私治理。没有真机基线时，不承诺百分比收益，也不扩大优化范围。

PR 顺序

六个性能包，预计 17-24 人日

编号独立于 7 月 11 日治理计划：PERF-0 可并行；PERF-1 兼容治理 PR-2；PERF-3、PERF-2、PERF-4 分别受治理 PR-8、PR-9、PR-10 约束；PERF-5 作为治理计划显式增补项，不得静默插队。

PERF-5

有界分片书籍仓储

语义章节 + 有界物理 chunk、ImportJob、feature flag、双读与 legacy shadow write。迁移包不删除旧数据，清理另立后续 PR。

5-7 人日

验收闸门

每个性能包都必须用行为证据退出

静态分析和单元测试只能证明代码质量，不能替代真机性能。Profile 结果、竞态用例和迁移回归缺一不可。

G0 基线可信

指定设备与场景可重复执行；startup trace 单独输出首帧与可交互时间，FrameTiming 分别输出 buildDuration、rasterDuration P95/P99、掉帧率和内存峰值，并签署目标及最大允许回退值。

G1 行为不回归

TTS 不重复、不跳句；2048 规则不变；默认书与本地书进度可恢复；隐私同意前网络为 0。

G2 资源有上限

runner、buffer、HttpClient、播放器、Timer、订阅和缓存文件数量均有可断言上限。

G3 帧预算达标

buildDuration 与 rasterDuration P95 分别低于设备真实帧预算；任一阶段超预算即计为慢帧，P99 与掉帧率不得超过已签署回退值。

G4 大书可取消

15MB UTF-8/GBK 导入可取消，取消后 Future、Isolate、端口与临时文件全部收口。

G5 版本可回滚

feature flag 可关闭新仓储；首个稳定版本双读并 legacy shadow write，使旧 APK 可读取新导入书。旧数据清理必须另立后续 PR。

风险管理

最高风险不在动画，而在异步生命周期与数据迁移

风险等级表示实施审慎程度，不代表当前已发生故障。每项风险都绑定暂停条件与回滚方式。

性能改进实施风险、控制措施与回滚条件

完成定义

什么情况下可以对外说“性能改进完成”

不是代码合并，也不是 analyze 通过，而是在目标设备上持续满足帧预算、资源上限和核心行为三类证据。
