EXECUTION CONTRACT

先定边界，再写代码

本详设服从 2026-07-11 全局治理计划。性能改造不能越过隐私闸门、架构边界和 TTS 两步下载契约，也不能靠拆文件或降低动效掩盖真实问题。

00 / SCOPE

测量先行

PERF-0 只建立真机 Profile 证据。所有收益百分比、P95 和内存峰值在采集前必须为 null。

帧预算分开算

buildDuration 与 rasterDuration 分别统计 P50/P95/P99；慢帧为任一阶段超预算，禁止相加伪造“总帧耗时”。

职责不扩张

tts_engine_service.dart 已 697 行，超过 service 警戒线；square_board.dart 已 800 行。两者只允许搬出职责和适配，不得继续塞入新模块。

数据纯本地

书籍正文、章节、游标和阅读进度不上传。TTS 仍严格执行 POST 获取 JSON URL，再 GET 下载音频。

回滚不越隐私

功能开关可以退回旧渲染、旧读取或旧调度；任何开关都不得恢复隐私同意前的 Sentry、网络、Provider 或音频初始化。

GameEngine 不上 Isolate

4 × 4 矩阵计算应保持纯 Dart 同步确定性；Isolate 只用于大文件解析、迁移等超过 100KB 的重任务。

Coordinator 接口与状态机
typedef AudioSettingsSnapshot = ({
  bool enabled,
  double volume,
  String style,
});

enum AudioStartupPhase { idle, scheduled, initializing, ready, degraded, disposed }

final class PostConsentAudioCoordinator {
  AudioStartupPhase get phase;

  Future<void> scheduleAfterFirstFrame(AudioSettingsSnapshot settings);
  Future<void> updateSettings(AudioSettingsSnapshot settings);
  Future<void> handleLifecycle(AppLifecycleState state);
  Future<void> dispose();
}
idle→ 首帧完成scheduled→initializing→ready
initializing→ 单项失败degraded→ 可显式重试ready
任意非终态→ dispose / 代际失效disposed
  • updateSettings() 只保留最新快照；初始化完成后应用一次，快速拖动音量不排队执行旧值。
  • 初始化中进入后台时记住 paused 目标态；完成后不得自动播放，恢复前台后才应用最新设置。
  • SFX 失败不阻断 Ambient，Ambient 失败不阻断 Dashboard；异常用 CyberLogger.captureWarning(tag: 'audio')。

SnapshotWriter 接口

enum SnapshotFlushStatus { noPending, persisted, failed }

final class SnapshotWriter<T> {
  SnapshotWriter({
    required Future<void> Function(T value) persist,
    Duration debounce = const Duration(seconds: 1),
    Duration? maxWait,
  });

  void submit(int revision, T value);       // 只接受更高 revision
  Future<SnapshotFlushStatus> flush();     // 暂停 / dispose 前调用
  void discardPending();
  Future<void> dispose();                  // flush 后终止
}

实现只允许一个 in-flight write。写入期间的新值保存在 pending；当前写完后只提交最高 revision。失败记录一次脱敏 warning 并保留最新 pending，等待下一次 submit/flush 重试，不做无界自旋。游戏使用 1 秒 debounce；阅读进度使用 750ms debounce + 5 秒 maxWait。

final class TtsRunnerCoordinator {
  int get currentEpoch;
  bool isCurrent(int epoch);

  Future<int> restart();       // 等旧 1+1 runner 退出后返回新 epoch
  Future<void> waitForItem(int epoch);
  Future<void> waitForSpace(int epoch);
  void signalItemReady();
  void signalSpaceReady();
  Future<void> stop();
}

Future<void> _playRunner(int epoch) async {
  while (_runnerCoordinator.isCurrent(epoch) && !_disposed) {
    if (_buffer.isEmpty) await _runnerCoordinator.waitForItem(epoch);
    if (!_runnerCoordinator.isCurrent(epoch)) return;
    await _playNext();
    _runnerCoordinator.signalSpaceReady();
  }
}

预取轨在 buffer 满时等待 spaceReady，播放轨在 buffer 空时等待 itemReady；停止触发 stopped 使所有等待立即退出。网络错误重试可使用可取消退避，但空闲态不得保留固定周期轮询。任意时刻断言 prefetch/play runner 各不超过 1。

RealHttpClient 由单一 owner 长期持有，只在 dispose 时 close(force: true)。每次 attempt 创建新 operationId；token 取消时 abort 当前 request、取消响应流、关闭 sink 并删除 .part。取消不计入连续失败，也不能触发本地 TTS 降级。

  Future<BookManifest?> loadManifest(String bookId);

关键执行规则

  • 主 Isolate 只把输入路径、staging 路径和 jobId 传给 worker；worker 流式解码并直接写 chunk。
  • 控制端使用独立 result/error/onExit port 和唯一 completeOnce；协作 cancel 超时后才 kill，kill 后也必须完成 Future 并关闭全部端口。
  • chunk、manifest、catalog 和 shadow 都先写 .part，校验后逐文件 rename；v2 与 legacy 都发布成功后才让书籍出现在书架。
  • Reader 默认缓存当前及相邻最多 3 个 chunk，建议总内存预算 768KB；实际阈值由 PERF-0 验证。
  • 删除或清理只按 manifest/job ledger 枚举明确文件逐个删除，再删除已空目录；禁止递归删除未知目录。

回滚红线：只要首个稳定版本仍承诺旧 APK 回滚，书籍必须在 legacy shadow 原子完成后才标记导入成功。若 shadow 失败，保留 v2 staging 供重试，但不向用户展示“已完成”。关闭 v2 read 不删除任何 v2 数据。

首个稳定版本不批量迁移旧书，也不在启动时迁移。旧书继续通过 dual-read fallback 使用原 JSON；只有新导入书同时生成 v2 与 legacy shadow。旧书主动迁移、停止 shadow 和清理 legacy 都必须在至少一个稳定版本后另立 PR，失败时继续读 legacy，永不删除旧数据。

跨包连接规则

Reader → TTS

Reader 提供句子源与 contentRevision；TTS 建立自己的 semanticSession。Reader 不接触播放器或 HTTP。

Settings → Audio

Settings 只发布不可变快照；AppComposition effect 调用 PostConsentAudioCoordinator。

Game → TTS

Game 发出用户活动领域事件；composition 连接到 TTS idle reset，GameProvider 不读取 engine service。

Library → Reader

Library 提供 BookContentRepository；Reader 请求有界 BookWindow，不读取具体文件或整书 JSON。

最小灰度方式

  1. 开发构建：开新写、关新读，比较 v2 与 legacy 内容。
  2. 内部构建：开新读并保留 fallback 与 shadow，记录本地计数，不上传正文或进度。
  3. 候选发布：默认启用单一性能包，完成 24 小时回归和旧 APK 演练。
  4. 稳定发布：只有该包退出门签署后才开始下一包默认启用。
